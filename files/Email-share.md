#### 邮件欺骗与网络钓鱼

- 电子邮件是网络钓鱼攻击的主要媒介 。

- **邮件欺骗 (Email Spoofing)：** 冒充可信发件人（如 CEO、管理员）的技术 。

现有防御机制：

- SPF： 检查发件人 IP 是否被授权 。

- DKIM： 通过加密签名验证邮件未被篡改 。

- DMARC： 统一 SPF/DKIM，并检查“From”字段的域对齐 。

> 如果这些防御都通过了，是否还存在威胁？

## 2. 核心漏洞

<!-- _header: \ ***![#l h:40](../images/logo.png)*** *背景* **漏洞** *实验 1* *威胁放大* *实验2* *源头追踪* *解决与讨论*-->
<!-- _class: navbar cols-2 -->

<div class="ldiv">

#### SMTP 走私 (SMTP Smuggling)

> 定义： 一种新型的邮件欺骗技术，可以绕过 SPF 和 DMARC 认证 。

- 核心原理：利用“不一致性”
- 发送方 (Sending MTA)： 只认标准结束符 `\r\n.\r\n`。
- 接收方 (Receiving MTA)： 为了兼容性，也认非标准结束符 (如 `\n.\n`) 。

</div>
<div class = "rdiv">

#### 攻击流程

1. 攻击者在邮件正文中嵌入“非标准结束符” `(\n.\n)` + 新的 SMTP 命令 。
2. 发送方“粗心”地将其视为一封邮件发出。
3. 接收方“上当”，将一封邮件错误地解析为两封。
4. 第二封“走私”的欺骗邮件成功绕过 SPF/DMARC 验证。

## 4. 威胁放大

<!-- _header: \ ***![#l h:40](../images/logo.png)*** *背景* *漏洞* *实验 1* **威胁放大** *实验2* *源头追踪* *解决与讨论*-->
<!-- _class: navbar cols-2-46 -->

<div class="ldiv">

SMTP 走私漏洞其最大的威胁在于，它可以和“共享 SPF 基础设施”相结合 ，允许攻击者利用一个免费的公共邮箱账户成功冒充成千上万个其他高信誉度的域名
**什么是共享 SPF？**

- 许多组织（youtube.com 为例）为了方便，会委托`第三方邮件服务商`（如 Google）来代表它们发送邮件
- 它们通过在自己的 SPF 记录中 **include** 谷歌的 SPF 记录来实现这一点

- **结果：** 谷歌的邮件服务器 IP 地址被同时授权代表 `gmail.com` 和 `youtube.com` 发送邮件

</div>
<div class = "rdiv">

#### 威胁模型如何运作？

1. 攻击者注册一个免费的 Gmail 账户（已知其发送端是脆弱的）
2. 目标接收方（如 victim@daum.net）的接收端是脆弱的

3. 攻击者从他的 Gmail 账户发起 SMTP 走私攻击，但在“走私”的第二封邮件中，他将发件人设置为 **admin@youtube.com**

4. 接收方 daum.net 收到这封伪造的 admin@youtube.com 邮件。它会检查 SPF：发现邮件来自谷歌的 IP 地址 。它查询 youtube.com 的 SPF 记录，发现 youtube.com 也授权谷歌的 IP 地址发信
**成功攻击!**
</div>

## 5. 实验 2：私有电子邮件服务实验

<!-- _header: \ ***![#l h:40](../images/logo.png)*** *背景* *漏洞* *实验 1* *威胁放大* **实验2** *源头追踪* *解决与讨论*-->
<!-- _class: navbar cols-2-46 -->

<div class ="ldiv">

> （第 4 章）解决了一个核心难题：研究人员如何能在不注册账户、不干扰正常用户的情况下，合乎道德地测试那些他们无法进入的私有邮件系统（如大学、企业）？

#### 阶段一：用户研究 (User Study)

这是最基础的“Ground Truth”收集阶段，通过招募内部人员来获取一手数据

- 确认私有邮件服务（以大学为例）是否存在漏洞
- 为后续的自动化测试方法收集“标准答案”，以验证其准确性 。
</div>

<div class ="rdiv">

**招募与伦理：** 研究获得了机构审查委员会 (IRB) 的批准 。他们招募了来自 48 所大学的参与者，并获得了他们的知情同意

**测试：** 研究人员向每位参与者的大学邮箱发送了 13 封测试邮件，每封包含一种不同的“走私暗号”（Payloads）。

**结果判断：**

如果攻击成功（邮件被一分为二），第二封“走私”邮件的标题会被设为 `“Successful! Please report it to us.”` 。

如果攻击失败，参与者收到`“Normal; Please Ignore.”`

**反馈：** 参与者通过提交在线问卷 。

**结果：** 在 48 个被测试的大学邮件服务中，有 23 个 被证实存在漏洞 。

缺点： 这种方法依赖人工，难以规模化，且用户报告可能出错 。

</div>

#### 阶段二：DKIM 侧信道

为了解决“用户研究”的缺点，研究人员验了一种更高级的**自动化**探测技术。

**核心思想:** 利用 DKIM（邮件签名）验证时，接收方服务器会查询 DNS 的特性，来充当“信使” 。

#### 阶段三：大规模非侵入式测试

这是最精妙的一步。虽然 DKIM 方法是自动的，但它仍然会向目标（包括无漏洞的）发送测试邮件 。为了进行更大规模（如 Tranco Top 10,000 域名）的测试，研究人员设计了一种 **完全“非侵入式”** 的方法 。

**目的：** 只在对方脆弱的情况下才发送邮件；如果不脆弱，则不留任何痕迹 。

**核心思想：**

- 研究人员构造一封特殊的测试邮件，邮件正文（蓝色部分）是一份完整的“漏洞报告” 。

- 在“报告”末尾，他们附加了“走私暗号”（如 \n.\n）和新的 SMTP 指令（红色部分）。

- 最关键的一点： 这封邮件故意不包含那个标准的、正确的邮件结束标记（\r\n.\r\n）。

## 6. 源头追踪：漏洞为何如此普遍？

<!-- _header: \ ***![#l h:40](../images/logo.png)*** *背景* *漏洞* *实验 1* *威胁放大* *实验2* **源头追踪** *解决与讨论*-->

> **核心问题：** 为什么在公共和私有邮件服务中，SMTP 走私漏洞都如此普遍？研究人员发现，这并非成千上万个服务各自犯了错，而是因为电子邮件生态系统`高度集中`，问题出在了几个核心的基础设施上 。

漏洞主要集中在三种类型的基础设施上：邮件安全网关、开源邮件软件和大型邮件服务商

- 第一大来源 (网关)：Proofpoint。这是一个本应提供安全防护的邮件安全网关，但它却与 512 个脆弱域名相关联

- 第二大来源 (软件)：Postfix。这是一个极其流行的开源邮件软件，与 316 个脆弱域名相关联

## 6. 源头追踪：漏洞为何如此普遍？

<!-- _header: \ ***![#l h:40](../images/logo.png)*** *背景* *漏洞* *实验 1* *威胁放大* *实验2* **源头追踪** *解决与讨论*-->
<!-- _class: navbar cols-2-55 -->

2. **邮件安全网关 (TrendMicro, Sophos, Proofpoint 等)：**

- **TrendMicro**（趋势科技）在接收端被证实是**脆弱的** 。
- **Sophos** 被证实是安全的 。
- **Proofpoint (推断)：** 由于难以获得测试账户，研究人员通过数据推断：在所有使用 Proofpoint 的域名中，有 **91.1%** 被证实是脆弱的，这强烈暗示了漏洞源于 Proofpoint 自身

> 论文针对 SMTP 走私漏洞，从“接收方”和“发送方”两个角度给出了明确的修复建议。

#### 1. 对接收方 (Receiving MTA) 的建议

- **根本解决方案：严格遵守标准**
  - 接收邮件服务器(MTA)应严格执行 RFC 5321 标准，**只承认 `\r\n.\r\n`** 作为唯一的 DATA 命令结束标记 。
  - _挑战_：困难在于更新那些为了兼容性而故意接受 `\n` 等非标符号的“遗留系统” 。RFC 标准其实早已警告过这种做法会带来问题
- **短期解决方案：禁用流水线机制 (Disabling Pipeline)**
  - 攻击者利用 SMTP 命令流水线机制来注入第二封（走私的）邮件 。
  - 禁用此功能可以作为一种短期缓解措施 ，Postfix 也提供了此选项 。
  - _缺点_：这会**牺牲邮件通信的性能和吞吐量** 。

#### 2. 对发送方 (Sending MTA) 的建议

- **首选解决方案 (推荐)：数据编码 (Data Encoding)**
  - 这是为了解决“数据”与“命令”无法区分的根本问题 。
  - **原理：** 将整个邮件正文内容使用 **Base64** 算法进行编码 。
  - **好处：** 编码后的数据（一长串无意义字符）**绝对不可能**被误认为是 SMTP 命令 。此方法完全符合 RFC 标准且兼容现有系统 。
- **短期修复：点填充 (Dot Stuffing) 和 字符串过滤**
  - _点填充_：在行首的 `.` 前再加一个 `.`，这是 RFC 5321 规范的操作 。
  - _字符串过滤_：过滤掉可疑字符（如 `\r`, `\n`, `\x00`）。
  - _缺点_：这些**不是根本的修复方法** 。实验证明（第 3.2 节），它们可以被**空字符 (`\x00`)** 绕过，因为 `\x00` 会干扰服务器的行识别 。

#### 什么是 Base64 算法？(及其工作原理)

**通俗解释：** **Base64** 是一种 **数据编码算法** 。它就像一个“翻译器”，可以把 **任意内容**（包括文本、图片、甚至攻击者插入的 `\n.\n` 和 `\x00` 等特殊字符）转换成一长串**安全的纯文本字符**。这个“安全”的字符集只包含 64 个字符`（A-Z, a-z, 0-9, + 和 /）`。

**它如何解决 SMTP 走私问题：**

SMTP 走私的根源在于，接收方服务器把邮件正文（数据）误认为是 SMTP 指令（命令）。Base64 正好可以解决这个问题：

1. **发送时编码:** 攻击者的恶意指令 `\n.\n MAIL FROM...` 就变成了一堆看似无意义的“乱码”，例如 `Cg4KIE1BSUwgRlJPTS4uLg==`。

2. **传输时（安全）：** 接收方服务器在处理 SMTP 指令时，看到的是这串“乱码”。它**完全无法**从这串字符中识别出 `\n.\n` 或 `MAIL FROM` 等指令，因此它只会把这串“乱码”当作普通的邮件正文数据来处理。**攻击就失效了**。

#### 伦理考量 (Ethics Considerations) 概要

研究人员在实验中严格遵守了伦理规范，其核心是**将干扰降到最低**：

- **IRB 批准：** 整个研究在开始前已获得**IRB（机构审查委员会）的审查和批准** 。
- **公共服务测试：** 均使用研究人员**自有账户**进行，并严格限制了发信频率（每 30 秒 1 封），避免了对服务的干扰
- **私有服务测试（关键）：**
  - “用户研究”获得了参与者的**知情同意**
  - “大规模非侵入式测试”设计得非常巧妙：如果目标不脆弱，则**不发送任何邮件** ；如果脆弱，则仅向管理员发送**一封**“漏洞报告”作为通知，这本身也是负责任披露的一部分
